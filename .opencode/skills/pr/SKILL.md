# PR — 提交 & 推送

提交本地更改、推送到远程，并**自动创建新 PR 或追加到已有 PR**

（根据分支状态自动判断）。

## 操作说明

- **严禁强制推送（force push）**。
- **严禁直接推送到 main/master** —— 必须走 PR 流程。
- 所有面向用户的文本必须使用用户在对话中使用的语言，**不得默认使用英文**。
- 提供选择时，使用带可选项的交互式对话框，而非纯文本。

---

## 步骤 1：前置检查

1. 执行 `git branch --show-current` —— 获取当前分支名。
    
    若为空（分离头指针），请用户先切换到正常分支，然后停止。
    
2. 执行 `gh pr list --head {branch} --state open --json number,url`
    
    查询当前分支是否存在**打开中的 PR**。
    

获取远程仓库的默认分支，记为 `{default_branch}`。

**模式判断**：

表格

|当前分支|该分支存在打开的 PR|执行模式|
|---|---|---|
|{default_branch}|不适用（需新建分支）|**创建**|
|其他分支|是|**追加**|
|其他分支|否|**创建**|

---

## 步骤 2：暂存并生成提交信息

执行 `git status --porcelain` 并检查输出：

- **无输出** → 无本地更改。直接跳到步骤 3（可能仍有未推送提交）。
- **既有已暂存，又有未暂存更改** → 询问用户：
    
    仅提交已暂存文件（**部分提交**），还是暂存所有更改（**完整提交**）？
- **其他情况** → 默认**完整提交**。

如果是 **完整提交**：执行 `git add -A`。

如果是 **部分提交**：跳过（文件已暂存）。

读取已暂存的差异（`git diff --cached`），并**按项目提交规范生成提交信息**。

若无规范，则默认生成**简洁英文**：

长度 ≤ 120 字符，以英文句点结尾，无多余标点，聚焦用户可感知的变更。

---

## 步骤 3：提交 & 推送

### 追加模式（Append mode）

如果有内容已暂存，先提交。

然后推送，并输出**已有 PR 的地址**。

如果无可提交、无可推送内容，告知用户并停止。

---

### 创建模式（Create mode）

#### 生成 PR 元信息

基于从 `{default_branch}` 以来的**所有变更**（暂存内容 + 已有提交），生成：

- **分支名**（仅在当前处于默认分支时生成）：
    
    遵循项目分支规范；若无，则使用
    
    `feature/{用户名}_主题` 或 `bugfix/{用户名}_主题`
    
    （`{用户名}` = 本地 Git 用户名，小写；通过 `git config user.name` 获取）。
    
    若当前已在非默认分支，直接使用当前分支名。
    
- **PR 标题**：遵循项目规范的简洁摘要；无规范则用简短英文句子。
    
    **必须基于从默认分支以来的所有变更**，而非仅最新提交信息。
    
- **PR 描述**：使用用户对话语言的纯文本（无 Markdown），
    
    简要说明**改了什么、为什么改**。
    

#### 创建分支、提交、推送

如果当前在 `{default_branch}`，先创建新分支：

`git checkout -b {branch_name}`

如果有暂存内容，执行提交。

推送并创建 PR：

plaintext

```
git push -u origin {branch_name}
gh pr create --title "{title}" --body "$(cat <<'EOF'
{description}
EOF
)"
```

向用户输出 **PR 地址**。
