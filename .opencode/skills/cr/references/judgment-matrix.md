# 判定矩阵

## 风险等级评估

风险等级按**单个问题**判定，而非按类型 —— 同一类别（如重命名）可能因范围和影响不同而属于低风险或高风险。

表格

| 风险等级 | 判定规则                   | 示例                                                                                                               |
| ---- | ---------------------- | ---------------------------------------------------------------------------------------------------------------- |
| 低风险  | 只有一种合理的修复方式            | 空值检查、修正错误注释、按规范重命名、删除冗余重复代码、添加 `reserve`、修复明显的越界 / 差一错误                                                          |
| 中风险  | 存在多种修复方式，但不涉及设计决策或外部契约 | 抽取函数间共享逻辑、删除未使用的内部方法、简化跨函数控制流、调整模块内部边界                                                                           |
| 高风险  | 涉及设计决策或外部契约            | 公有 API 变更（签名、行为、废弃）、测试基线变更、架构重构、有多种可行方案的算法替换、引入新依赖、修改数据持久化 / 序列化格式、修改线程模型、涉及时空权衡的性能优化、超出既定修复范围的用户可见行为变更、构建系统配置修改 |

## 按风险等级处理

表格

|`FIX_MODE` 修复模式|低风险|中风险|高风险|
|---|---|---|---|
|full（全量）|自动修复|自动修复|自动修复|
|low_medium（低中）|自动修复|自动修复|需用户确认|
|low（仅低）|自动修复|需用户确认|需用户确认|

**full 模式特殊规则**：

任何会修改测试基线（截图对比、基准文件）的问题，**无论风险等级如何，都必须交由用户确认**，不自动修复。

## 是否值得修复？

代码检查清单与文档检查清单定义**检查内容**，本节定义**发现问题后是否需要修复**。

### 决策原则

1. **必须修复** —— 问题影响运行时正确性、安全性或隐私安全。
2. **明确则修复** —— 问题可提升代码质量（性能、简化、架构）。仅当解决方案无歧义、不会引入新风险时才修复。性能修改必须同时满足：对语义等价性有极高把握，且收益大于代码复杂度增加的净收益。
3. **不一致则修复** —— 问题涉及命名、初始化、注释或文件组织。仅当违反项目内置规则，或与周边代码既定模式矛盾时修复。
4. **永远跳过** —— 纯风格偏好（不违反任何一致性规则）、基于假想未来需求而非当前代码的建议、对无正确性问题的稳定代码进行替代性重写。

### 例外情况

- 重复代码抽取：仅在逻辑明显完全重复时修复（不按行数阈值判断，按复杂度与维护成本评估）。
- 非 Bug 修复类的公有 API 签名修改：仅当对 API 使用者有明确收益时才修复，**永远属于高风险**。
- 测试覆盖率缺口与回归风险：**仅标记提醒，不自动修复**—— 上报给用户知晓即可。

## 反模式（禁止修复）

极易产生误报的模式。除非有强证据证明是真实 Bug，否则一律跳过：

- ** speculative optimizations 推测性优化 ** —— 构建系统微调、添加缓存、无条件防护判断等，无已证明的故障或可测量瓶颈。
- **文档示例 “简化”** —— 从出于教学目的而刻意详细的示例中删除属性、参数或步骤。这**不等同于**删除冗余代码。
- **伪装成 Bug 修复的行为变更** —— 如果提议的修复会改变可观测行为（而非仅实现细节），必须确认原行为确实是 Bug，而非有意设计。若仅从差异上下文无法确认意图，则只标记不修复。
