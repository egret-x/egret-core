# 团队评审

你是**协调者**。创建一个智能代理团队，并分派评审员、验证员和修复员代理。

这对多轮迭代至关重要：代理是用完即弃的，因此你的上下文只通过**结构化问题列表**增长，而不是原始文件内容 —— 避免因上下文压缩而丢失早期记录。**永远不要直接修改文件**。仅在仲裁和诊断时读取代码。

评审循环专为**不间断的多轮迭代**设计。

在涉及用户之前，务必先处理所有可自动修复的问题。在进入 ** 确认阶段（第 6 阶段）**或**报告阶段（第 7 阶段）** 之前，**不要暂停并向用户询问任何内容**。报告后的唯一交互是可选的检查清单演进提示。

**评审员–验证员对抗对**是核心质量机制：评审员发现问题，验证员对其提出质疑。这种双方校验能显著减少误报。

评审员和验证员**不得查看对方的输出或共享对话历史**。

## 来自 SKILL.md 的输入

- `FIX_MODE`：low | low_medium | full

## 流程

plaintext

```
范围确定
  │
  ↓
┌─────────── 评审循环 ───────────┐
│                                 │
│  评审 → 过滤 → 修复/验证        │
│    ↑                     │      │
│    └── 新问题 ───────────┘      │
│                                 │
└─────────────────────────────────┘
  │ 无新问题
  ↓
存在待处理/失败问题？ ──否──→ 报告 + 检查清单演进
  │ 是
  ↓
确认 ──全部跳过──→ 报告 + 检查清单演进
  │ 已批准
  ↓
修复/验证 ──→ 回到评审循环 ↑
```

只要发现新问题，评审循环就会重复执行。

每一轮都是**全新的完整评审**—— 而不是只针对之前修复内容的定向复查。

**确认阶段之后**：已批准的修复会进入修复 / 验证环节，然后**始终重新进入评审循环（第 2 阶段）**，以捕获修复引入的新问题。

只有当某一轮没有产生新问题时，循环才会退出到：

- 确认阶段（仍有待处理 / 失败问题）
- 或报告阶段（无任何遗留问题）

## 第 1 阶段：范围确定

根据 `$ARGUMENTS` 确定要评审的差异：

- **参数为空**：按顺序检查常用基础分支：`main`、`master`，使用第一个存在的分支。获取分支差异：
    
    plaintext
    
    ```
    git merge-base origin/{base_branch} HEAD
    git diff <merge-base-sha>
    ```
    
- **提交哈希**（如 `abc123`）：使用 `git rev-parse --verify` 验证，然后执行 `git show`。
- **提交范围**（如 `abc123..def456` 或 `abc123...def456`）：验证两端点，获取包含两端点的差异：
    
    plaintext
    
    ```
    git diff A~1..B
    ```
    
- **文件 / 目录路径**：验证所有路径在磁盘上存在，然后读取文件内容。

如果差异为空 → 显示用法示例并退出：

`/cr`（未提交变更或当前分支）、

`/cr a1b2c3d`、`/cr a1b2c3d..e4f5g6h`、

`/cr src/foo.cpp`、`/cr 123`、`/cr https://github.com/.../pull/123`。

### 关联 PR 评论（可选，尽力而为）

如果可用 `gh` 命令，检查当前分支是否存在开放 PR：

plaintext

```
gh pr view --json number,state --jq 'select(.state == "OPEN") | .number' 2>/dev/null
```

如果存在开放 PR，获取其行级评审评论：

plaintext

```
gh api repos/{owner}/{repo}/pulls/{number}/comments
```

存储为 `PR_COMMENTS`，供评审阶段校验使用。

### 构建基线

仅文档类变更则跳过。

如果无法确定构建 / 测试命令，发出警告：修复验证将被跳过。

否则执行构建 + 测试。构建失败 → 终止流程。

读取基础分支以来的 Git 日志，获取近期修复上下文（避免重复标记之前 `/cr` 已修复的问题）。

### 模块划分

将范围内的文件划分为**评审模块**以支持并行评审。

每个模块是自包含的逻辑单元。

大文件按章节 / 函数组拆分；相关小文件归为一组。

将每个模块归类为 `code`、`doc` 或 `mixed`。

### 持久化状态

初始化 `CR_STATE_FILE`：若不存在则创建 `.cr-cache/`。

文件名从评审范围派生：`.cr-cache/{branch}.md`（将 `/` 清洗为 `-`，例如 `feature/dom_text_box` → `feature-dom_text_box.md`）。

每次 `/cr` 调用都是**全新会话**。

永远不要从已有文件读取或恢复 —— 每次会话都从头开始全新完整评审。

如果派生文件名已存在（上一次或并发会话残留），使用最小未使用数字后缀（`-2`、`-3`…）。

将选定路径记录为 `CR_STATE_FILE`。

将会话信息（模式、阈值、文件列表、模块分配、变更行范围、构建 / 测试命令）和增量更新的问题列表写入 `CR_STATE_FILE`。

`CR_STATE_FILE` 由协调者独占管理 —— 团队代理永远不读写该文件。

将所有无法解决的内容记录到 `CR_STATE_FILE` 供用户评审。

**`CR_STATE_FILE` 中的问题格式**：

markdown

```
# 问题列表

## 1. [简要描述]
- **状态**：pending | approved | fixed | failed | skipped
- **原因**：[无法自动修复的原因，例如“高于自动修复阈值（高风险）”、
  “修复失败：重试2次后仍构建错误”、“已回滚：引入回归问题”]
- **风险**：low | medium | high
- **文件**：[文件路径:行号]
- **现状**：[代码当前行为]
- **建议**：[修复将修改的内容——中/高风险需包含协调者选定的具体方案与理由]
- **影响**：[还会影响哪些内容]
```

状态说明：

- `pending` — 已记录，等待用户在第 6 阶段决策
- `approved` — 用户在第 6 阶段批准修复，已发往第 4 阶段
- `fixed` — 修复已应用并通过验证
- `failed` — 已尝试修复，但验证后失败
- `skipped` — 用户拒绝或问题已被驳回（不再重复上报）

---

## 第 2 阶段：评审

### 本轮不变规则

- **恢复状态**：每轮开始时协调者读取 `CR_STATE_FILE`。评审员不读取此文件。
- **已知问题排除列表**（第 2 轮及以后）：由协调者从 `CR_STATE_FILE` 提取 —— 每条仅包含文件路径、行范围和单行描述。无风险等级、无验证员结论、无修复结果。
- **跨轮去重**（协调者，第 3 阶段）：以 `CR_STATE_FILE` 为安全网进行去重。之前确认阶段被用户拒绝的问题也会被排除。**已修复问题不排除**—— 修复代码中出现的新问题属于有效新问题。

### 团队组建

为本轮创建全新团队。每轮使用新团队 —— 不重用前一轮代理（团队关闭后会丢失上下文）。

- 每个模块分配一个评审员代理（`reviewer-N`）。
- 一个**验证员**代理（`verifier`），所有模块共享。

### 评审员提示词

立场：**彻底**—— 尽可能发现真实问题，提交前自行校验。

每个评审员接收：

- **范围**：模块对应的文件列表 + 变更行范围。评审员按需自行获取差异和读取上下文 —— 协调者不传递原始差异或文件内容。
- **检查清单**：代码模块用 `code-checklist.md`，文档用 `doc-checklist.md`，混合模块两者都用。
- **证据要求**：每个问题必须附带当前代码树中的引用（文件：行 + 代码片段）。
- **已知问题排除**（第 2 轮及以后）：跳过匹配协调者排除列表的问题，专注发现新问题。
- **检查清单排除**：参见对应检查清单的排除章节。上下文中加载的项目规则优先级更高。
- **自检**：提交前重新阅读相关代码并校验每个问题，标记为确认或撤回。只提交确认无误的问题。如果引用的路径 / 行已不存在，通过 `git diff --name-only` 或文件搜索定位正确文件 / 路径后再上报。
- **输出格式**：`[文件:行号] [A/B/C] — [描述] — [关键代码行]`

**PR 评论评审员**（存在 `PR_COMMENTS` 时，仅第 1 轮）：额外一个代理，用于校验 PR 评审评论与当前代码是否一致。输出格式、校验流程相同。后续轮次跳过。

### 验证（流水线）

立场：**对抗性**—— 默认质疑评审员，主动寻找每个问题错误的理由。用真实证据驳回，成立则确认。

此步骤**强制执行**—— 协调者不得跳过或自行验证。

验证员以**流水线**方式运行 —— 不等待所有评审员完成。

每当评审员通过 SendMessage 发送报告，协调者必须**立即转发**给验证员。转发规则：

- **原样引用**：将评审员原始 SendMessage 内容用引用块包裹，原样发送。
- **不重写**：不总结、不重组、不合并多份评审报告、不添加协调者注释。
- **每份评审单独转发**：每个评审员报告作为独立消息发给验证员。

每条发送给验证员的提示词中**必须原样包含**以下内容：

plaintext

```
你是代码评审验证员。你的立场是对抗性的——默认质疑评审员的结论，主动寻找该问题可能不成立的理由。
你的工作是对每个问题进行压力测试，确保只有真实问题保留下来。

对每个收到的问题：

1. 读取引用代码（文件:行）及足够的上下文。
2. 主动尝试反驳该问题：评审员的推理是否有缺陷？是否存在使其成为非问题的上下文
   （如调用方保证的不变量、平台约束、有意设计）？代码实际行为是否如评审员所述？
   找出你能找到的最强反驳论据。
3. 每个问题输出：
   - 结论：REJECT 或 CONFIRM
   - 推理：REJECT 需说明具体反驳论据；CONFIRM 则简要说明你检查的内容及为何无有效反驳。

重要约束：
- 你的反驳必须基于代码中的真实证据。不要编造假想防御或不存在的调用方保证。
- CONFIRM 不是失败——它意味着评审员发现了真实问题，而你的挑战验证了这一点。
```

### 评审结束

保留所有评审员（在第 4 阶段复用为修复员），关闭验证员 → 进入第 3 阶段。

---

## 第 3 阶段：过滤 —— 仅协调者执行

进入此阶段前确认：

1. 所有评审员已提交最终报告；
2. 验证员已对所有上报问题给出 CONFIRM/REJECT 结论。
    
    **两条都满足才可继续**。

立场：**中立**—— 不信任任何单方。将评审员报告与验证员反驳视为同等权重输入。

使用项目全局视角考虑跨模块影响、规范和架构意图，这些是本地评审员可能忽略的。

### 3.1 去重

- 移除已在 `CR_STATE_FILE` 中的问题
- 移除评审员间重复问题（同一位置、同一主题）
- 移除之前确认阶段被用户拒绝的问题

已修复问题**不排除**—— 修复代码中出现的新问题有效。

### 3.2 存在性检查

表格

|验证员结论|动作|
|---|---|
|CONFIRM|合理性检查 —— 验证描述与引用代码一致。如有疑问则阅读代码。|
|REJECT|阅读代码。评估双方论据。仅当反驳论据可靠时才丢弃问题。|

### 3.3 风险等级

查阅 `judgment-matrix.md` 进行风险等级评估、是否值得修复、按风险处理及特殊规则。

**修复方案**（仅限中 / 高风险）：写明选定方案与理由，记录在问题的 `Proposed` 字段。

低风险：只有一种明显修复，无需额外说明。

### 3.4 路由 —— 记入 CR_STATE_FILE

所有确认通过的问题都会记录风险等级。

表格

|风险与 `FIX_MODE` 对比|→|
|---|---|
|等于或低于阈值|加入自动修复队列|
|高于阈值|记入 `CR_STATE_FILE` 并标记为 `pending`|

- 跨模块影响：如果修复需要更新修复员模块之外的内容，创建后续修复任务。
- 之前已回滚的问题：本轮不再尝试。

自动修复队列非空 → 第 4 阶段；队列为空 → 第 5 阶段。

**必须先处理所有可自动修复问题**—— 在全部自动修复完成并验证前，不得将 `pending` 问题展示给用户。

---

## 第 4 阶段：修复 / 验证

本阶段是评审循环和确认后流程复用的原子单元。

### 修复

立场：**精确**—— 完整正确地应用每个修复，绝不扩大范围。

协调者**不得直接执行修复**。

**代理分配**：可用时复用存活的评审员作为修复员 —— 每个评审员已具备所评审文件的上下文。

若无存活评审员（如第 5 阶段关闭团队后），创建新修复员代理。

协调者**必须按明确文件列表分配**，并确保同一文件同一时间只由一个修复员负责：

- 问题所在文件已被存活评审员读取 → 分配给该评审员
- 否则 → 创建修复员代理（跨模块问题用 `fixer-cross`）
- 多文件重命名 → 单个原子任务分配给一个代理

一个代理可接收多个修复任务（如果覆盖多个文件）。

避免将同一文件分配给多个代理，防止并发编辑冲突。

每个修复员接收：

- 问题描述 + 文件路径 + 行范围（修复员自行读取文件）
- 来自 `Proposed` 字段的修复方案（中 / 高风险）
- 修复员规则（**每条提示词必须原样包含**）：

plaintext

```
修复规则：
1. 每个问题修复后立即执行：git commit --only <文件> -m "提交信息"
2. 只修改协调者明确分配的文件。永远不要使用 git add .
3. 如果修复需要修改未分配文件，停止并上报协调者重新分配。
4. 提交信息：英文，不超过120字符，以句号结尾。
5. 有疑问时，宁愿跳过也不要冒险改错。
6. 不执行构建或测试。
7. 不修改公有 API 函数签名或类定义（注释可改），
   除非协调者问题描述明确要求修改 API 签名。
8. 每次修复后，检查变更是否影响分配文件内的相关注释或文档
   （函数/类文档注释、描述修改逻辑的行内注释）。
   如果影响，在同一提交中一并更新。
   跨模块文档更新（README、规范文件、其他模块）由协调者单独处理。
9. 完成后上报每个修复的提交哈希，以及所有跳过的问题和跳过原因。
```

每个修复员按问题提交（一个问题一次提交）。

### 验证

等待所有修复员完成。执行构建 + 测试。

- 无构建 / 测试命令或仅文档模块 → 跳过。
- **通过** → 在 `CR_STATE_FILE` 中将问题标记为 `fixed` → 第 5 阶段。
- **失败** → 二分定位失败提交，回滚，重新验证剩余内容，再追责（一个坏提交可能引发连锁失败）。
    
    对每个失败问题：使用原修复员代理重试（附带失败详情，最多重试 2 次），或回滚并标记为 `failed`。

---

## 第 5 阶段：是否继续？

### 步骤 1：关闭本轮团队

关闭团队以释放评审员和修复员。强制终止无响应代理。

### 步骤 2：路由

**按从上到下顺序匹配条件，执行第一个匹配项**。

表格

|条件|→|
|---|---|
|本轮确认并修复了任何问题|第 2 阶段（新一轮评审）|
|`CR_STATE_FILE` 中有 `pending` 或 `failed`|第 6 阶段|
|其他情况|第 7 阶段|

第一条规则含义：

如果本轮评审发现真实问题并已应用修复，**永远执行一次全新完整评审**，以捕获上一轮可能遗漏的问题。

这**不是**对修复内容的定向复查 —— 而是对**整个范围**的全新完整评审，使用全新团队。

---

## 第 6 阶段：确认

按风险（高 → 低）分组展示 `pending` + `failed` 问题，每组内按文件路径排序：

`[编号] [文件:行号] [风险] [原因] — [描述]`

以多选形式展示问题。每个选项标签为问题摘要（如 `[风险] 文件:行号 — 描述`）。

勾选 → `approved`，不勾选 → `skipped`。

如果用户回复批量指令（如 “全部修复”、“跳过剩余”），

仅应用于**等于或低于**当前 `FIX_MODE` 阈值的问题。

高于阈值的问题仍需单独确认。

- **全部跳过** → 第 7 阶段。
- **有任何批准** → 第 4 阶段（修复 / 验证）。验证后进入第 5 阶段（是否继续？）。
    
    若发现新问题，重新进入完整评审循环（第 2 阶段）。
    
    若无新问题但仍有 `pending`/`failed`，返回本阶段（第 6 阶段）。
    
    若无任何遗留问题，进入第 7 阶段。

---

## 第 7 阶段：报告

强制终止所有仍在运行的代理。关闭团队。

汇总：

- 轮次与每轮统计
- 问题发现 / 修复 / 跳过 / 失败数量
- 已回滚问题及原因
- 最终测试结果
