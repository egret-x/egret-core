# 代码评审检查清单

按优先级顺序评审：A（影响最高）→ B → C。

评审员提示词会指定需要检查哪些等级。

测试代码：仅检查明显的实现错误。

上下文加载的项目规则优先于本清单。

---

## A. 正确性与安全性

直接影响运行时行为的问题。

### A1. 代码正确性

- 返回值 / 输出参数在所有分支（包括错误分支）均正确设置
- 实现与函数名 / 注释描述的行为一致
- 条件逻辑无 && / || 混淆、缺少取反、优先级错误
- switch/case 覆盖所有分支，无意外穿透

### A2. 边界条件

- 除零保护（浮点与整数均需）
- 空容器在调用 front () /back () /operator [] 前已检查
- 空值 /nil/undefined 解引用已防护
- 整数溢出 / 下溢已处理（尤其无符号减法）
- 数组 / 字符串越界已检查

### A3. 错误处理

- I/O 操作结果已检查错误
- 解析结果在使用前已校验
- 外部输入已校验合法性
- 调用失败时有合理降级 / 安全返回
- Promise / 异步调用已正确等待并处理错误

### A4. 注入与敏感数据

- 用户输入在插入 DOM 前已过滤（innerHTML、dangerouslySetInnerHTML、v-html、[innerHTML]、document.write 等）
- URL 参数、localStorage、postMessage 数据在使用前已校验
- 客户端代码无硬编码 API 密钥、令牌或凭证

### A5. 资源管理

- 手动分配的资源在所有路径上均释放（优先使用 RAII）
- 文件句柄 / 系统资源已正确关闭
- 锁的获取与释放正确配对
- 数据库连接 / 网络套接字在 finally /defer 块中释放
- 事件监听、定时器、订阅、观察者在卸载 / 作用域退出时已清理

### A6. 内存安全

- 无移动后使用（被移动对象未重复使用）
- 无垂悬引用 / 局部变量指针
- 容器修改后不继续使用旧元素引用

### A7. 线程安全

> 仅在访问模式明显不安全时标记。

- 共享可变状态访问已加锁或使用原子操作
- 回调 / 闭包未捕获生命周期可能早于调用结束的引用或指针
- 条件变量等待带谓词（避免虚假唤醒）

### A8. 公有 API 注释

- 公有 API 注释准确描述当前行为、参数、返回值
- 注释中的取值范围、约束、错误条件与实现一致
- API 行为变更时注释同步更新
- 公有 API 有充足的参数与返回值说明
- 代码本身无法表达设计意图时已添加说明

## B. 重构与优化

提升代码质量、性能与可维护性的改进。

### B1. 性能

- 大小可预测的容器已预先分配空间
- 无不必要的深拷贝（仅在语义等价确定时标记）
- 循环不变表达式已提到循环外
- 循环内频繁字符串拼接已优化
- 无不必要的临时对象构造
- 大对象以 const& 传递而非值传递
- 无因缺少缓存、引用不稳定或 Props 内联对象 / 函数导致的不必要重渲染
- 未全量引入大型依赖（仅使用小部分时）

### B2. 代码简化

- 明显重复或相似逻辑已抽取（按复杂度与维护成本判断，不按行数阈值）
- 深层嵌套 if/else 用提前返回简化
- 冗余条件检查已合并或消除
- 过长函数拆分为单一职责子方法

### B3. 模块架构

> 仅当差异引入新依赖或跨模块移动代码时标记。

- 模块职责清晰，无边界越界
- 依赖方向合理
- 无循环依赖

### B4. 接口使用

- 调用的 API 符合设计意图与文档
- 未使用已废弃接口

### B5. 接口变更

> 仅标记 —— 描述变更与范围，交由协调者评估。

- 识别并描述公有 API 签名或类接口变更

### B6. 测试覆盖率

> 仅标记 —— 上报提醒，不自动修复。

- 变更的逻辑路径有对应测试用例
- 边界条件有测试覆盖
- 错误路径有测试覆盖

### B7. 回归风险

> 仅标记 —— 上报提醒，不自动修复。

- 已评估修改对其他调用方的影响
- 行为在所有目标平台保持一致

### B8. 渲染正确性

- 列表项使用稳定、唯一 key（非数组下标）
- 副作用正确放在生命周期钩子 /useEffect 并带正确依赖数组
- 组件状态推导正确（无过期闭包、无不同步派生状态）

## C. 规范与文档

编码规范与文档一致性。

### C1. 项目规范

- 命名遵循项目现有风格（按加载的项目规则）
- 变量名语义清晰，无不必要缩写
- 新代码命名与同文件风格一致
- 变量声明时赋初值（按项目规则）
- 类成员变量在声明或构造函数中初始化
- 代码符合项目规则中的语言使用限制
- 新代码与项目现有模式一致

### C2. 文件组织

- 实现文件中函数顺序与声明一致
- 头文件有合适的头文件保护
- 包含 / 导入依赖必要且合理

### C3. 类型安全

- 无隐式窄化转换（大类型 → 小类型）
- 无有符号 / 无符号混合比较
- 魔法数字抽取为具名常量（上下文已明确含义除外）

### C4. 常量正确性

- 不修改状态的方法标记为 const
- 未修改参数以常量引用传递
- 未修改局部变量声明为 const

### C5. 文档一致性

- 代码中的类型名 / 枚举名与项目文档一致
- 注释中的取值范围与规格文档一致

### C6. 可访问性

- 图片有意义的 alt 文本（装饰性图片为空 alt）
- 表单输入有关联标签
- 可交互元素支持键盘导航并使用语义化 HTML

## 排除列表

> 项目规则优先于本排除列表。若项目规则对某类被排除问题有明确要求，则该类**不被排除**—— 按项目规则评审。

1. 属于格式化工具范围内的纯风格偏好（非项目规则强制要求）
2. 项目格式化工具已处理的格式问题（缩进、空白等）
3. 基于假想未来需求、而非当前代码的建议
4. 代码遵循项目现有风格，但不符合某外部标准
5. 测试代码中的 C 级优先级问题（项目规则另有要求除外）
6. 对现有稳定、无错误代码的 “更好替代方案” 类建议
