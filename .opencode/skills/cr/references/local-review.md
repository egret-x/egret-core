# 本地评审

针对本地变更的**单代理评审**。对代码差异进行评审，展示确认后的问题，并让用户交互式选择需要修复的项。

## 参考文件

表格

|文件|用途|
|---|---|
|`code-checklist.md`|代码评审检查清单|
|`doc-checklist.md`|文档评审检查清单|
|`judgment-matrix.md`|风险等级、是否值得修复的判定标准、特殊规则|
|`checklist-evolution.md`|检查清单更新流程与规则|

---

## 步骤 1：范围确定

根据 `$ARGUMENTS` 和工作树状态，确定需要评审的差异范围：

- **`$ARGUMENTS` 为空**，**存在未提交变更**：
    
    范围仅为未提交变更。使用 `git diff HEAD` 获取（已暂存 + 未暂存的已追踪文件）。
    
    同时通过 `git status --porcelain` 检查未追踪文件（`??` 行），并读取其内容参与评审。
    
- **`$ARGUMENTS` 为空**，**无未提交变更**：
    
    按顺序检查常用基础分支：`main`、`master`，使用第一个存在的分支。
    
    获取分支差异：
    
    plaintext
    
    ```
    git merge-base origin/{base_branch} HEAD
    git diff <merge-base-sha>
    ```
    
    同时通过 `git status --porcelain` 检查未追踪文件（`??` 行）。
    
- **提交哈希**（如 `abc123`）：
    
    使用 `git rev-parse --verify` 验证，然后执行 `git show`。
    
- **提交范围**（如 `abc123..def456` 或 `abc123...def456`）：
    
    验证两端点，获取包含两端点的差异：
    
    plaintext
    
    ```
    git diff A~1..B
    ```
    
- **文件 / 目录路径**：
    
    验证所有路径在磁盘上存在，然后读取文件内容。
    

如果差异为空 → 显示用法示例并退出：

`/cr`（未提交变更或当前分支）、

`/cr a1b2c3d`、`/cr a1b2c3d..e4f5g6h`、

`/cr src/foo.cpp`、`/cr 123`、`/cr https://github.com/.../pull/123`。

---

## 步骤 2：评审

对差异进行评审。

代码文件使用 `code-checklist.md`，文档文件使用 `doc-checklist.md`。

当变更行依赖上下文时，按需读取相关代码段或关联定义。

未追踪文件没有差异 —— 将其完整内容作为新代码进行评审。

对每个发现的问题：

- 提供当前代码树中的引用（文件：行号 + 代码片段）。
- 通过重读代码进行自校验 —— 确认或放弃该问题。
- 如果引用的路径 / 行已不存在，在上报前通过 `git diff --name-only` 或文件搜索定位正确文件 / 路径。

**输出规则**：只向用户展示最终确认后的问题。

不输出分析过程、排除理由，或仅被考虑但已排除的问题。

---

## 步骤 3：过滤

查阅 `judgment-matrix.md` 进行**风险等级评估**、是否值得修复及特殊规则判断。

丢弃不值得上报的问题。

如果过滤后无任何问题 → 输出「未发现问题」并退出。

---

## 步骤 4：报告与修复

展示本次评审内容的摘要，然后列出所有确认后的问题。

通过**单次多选提问**让用户选择需要修复的项，每个选项标签为问题摘要（如 `[风险] 文件:行号 — 描述`）。

用户在一次提示中可勾选多个问题。

如果用户选择了任意问题，则应用对应的修复。

---

## 步骤 5：检查清单演进

回顾本次会话中所有确认后的问题。

如果存在当前检查清单未覆盖的**重复出现模式**，则读取 `checklist-evolution.md` 并按流程执行更新。
