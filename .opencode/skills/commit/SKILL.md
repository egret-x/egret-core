# 提交 — 仅本地提交

将本地更改提交至当前分支。不推送代码，也不创建拉取请求（PR）。

若当前处于默认分支，则先创建新分支再提交。

## 操作说明

- **禁止**直接在默认分支（main/master）上提交代码 —— 务必先创建新分支。
- **禁止**推送代码。本操作仅创建本地提交。
- 所有面向用户的文本必须使用用户在对话中使用的语言，不得默认使用英语。

---

## 步骤 1：前置检查

执行 `git branch --show-current` —— 获取当前分支名称。

若返回空值（处于分离头指针状态 /detached HEAD），请告知用户先切换到有效分支，然后终止操作。

确定远程仓库的默认分支，并将其记为 `{default_branch}`。

若当前正处于 `{default_branch}` 分支，需先创建新分支再提交：

- 若项目有分支命名规范，遵循规范生成分支名；若无，则使用 `feature/{用户名}_主题` 或 `bugfix/{用户名}_主题`（`{用户名}` = 本地 Git 用户名，小写；通过 `git config user.name` 获取）。
- 执行 `git checkout -b {branch_name}` 创建分支。

---

## 步骤 2：暂存 & 提交

执行 `git status --porcelain` 并检查输出结果：

- **无输出** → 无本地更改。告知用户并终止操作。
- **存在已暂存更改，无未暂存更改** → 直接执行提交。
- **存在已暂存更改，同时有未暂存更改** → 询问用户：仅提交已暂存的文件（「部分提交」），还是暂存所有更改（「完整提交」）？
    
    若选择「部分提交」：跳过暂存步骤（文件已处于暂存状态）。
    
    若选择「完整提交」：先执行 `git add -A`。
- **无已暂存更改，存在未暂存更改** → 执行 `git add -A` 暂存所有更改，然后提交。

读取已暂存的差异内容（`git diff --cached`），并遵循项目的提交信息规范生成提交说明。若未找到项目规范，则默认生成简洁的英文提交说明：长度不超过 120 个字符，以句点结尾，无其他标点符号，聚焦用户可感知的更改内容。

执行提交后，输出单行结果，格式如下：

`已提交 {短哈希值}：{提交说明}`

若用户选择「部分提交」，需额外告知用户：仍有未暂存的更改未纳入本次提交。

---

### 关键术语说明（确保技术准确性）

1. **detached HEAD**：分离头指针（Git 特有状态，保留专业术语 + 中文解释）
2. **staged changes**：已暂存更改（Git 标准译法）
3. **unstaged changes**：未暂存更改
4. **short_hash**：短哈希值（Git 提交哈希的简写形式）
5. **commit message**：提交说明（而非「提交信息」，更符合国内 Git 使用习惯）
6. **default branch**：默认分支（统一译法，替代「主分支」更精准）
7. **porcelain**：（参数保留英文，无通用中文译法，Git 官方文档亦保留该词）
