# 合并提交 — 整理分支历史

分析当前分支上的提交，并将其合并为清晰、逻辑完整的单元。

将目的相同的连续提交进行合并；重写不符合项目规范的提交信息。

在用户明确确认结果之前，**不会修改原分支**。

## 操作说明

- **严禁**在默认分支（main/master）上操作。
- 所有面向用户的文本必须使用用户在对话中使用的语言，**不得默认使用英文**。
- **步骤 1、2 和步骤 3 的 A 阶段，不得暂停或要求用户确认**，必须连续无中断执行。
    
    **但在 B 阶段，必须停下来向用户确认** —— 这是整个脚本唯一的强制交互点。详见 B 阶段。
- **在 Bash 调用中严禁使用 `$()`、反引号、shell 变量赋值** —— 沙箱会拒绝所有这类写法。
    
    每次 Bash 调用必须是**纯字面、独立完整**的命令，不允许动态展开。
    
    若需要把一个命令的输出用于另一个命令，先单独执行前一个命令，读取输出，
    
    再**手动把值拼进下一条命令**（即直接把值写死在命令文本里）。

---

## 步骤 1：确定提交范围

清理上一次中断运行残留的临时工作区或分支（匹配 `squash-tmp-{分支名}` 格式）。

根据 `$ARGUMENTS`，按**第一条匹配规则**解析出 `{squash_base}` 和 `{squash_end}`。

使用 `git rev-parse` 判断参数是分支名、提交哈希还是范围。

表格

|匹配模式|`{squash_base}`|`{squash_end}`|
|---|---|---|
|空（无参数）|远程追踪点（从未推送则回退到与默认分支的 merge-base）|HEAD|
|`branch`（关键字）|当前分支与默认分支的 merge-base —— 分析**整个分支**|HEAD|
|参数是分支名（如 `feature/foo`）|该分支与默认分支的 merge-base|该分支的 HEAD|
|参数是单个提交（如 `abc1234`）|`<提交>~1`（父提交，确保该提交本身被包含）|HEAD|
|参数是范围（如 `abc1234...def5678`）|`<较早提交>~1`（父提交，确保较早提交被包含）|`<较新提交>` —— 用 `git merge-base --is-ancestor` 判断顺序；用户写反则交换|

默认（空参数）情况：如果 HEAD 等于追踪点（无未推送内容），提示用户并停止。

用**单条**命令收集 `{squash_base}..{squash_end}` 范围内的提交：

plaintext

```
git log --first-parent --stat --format="%H %P %s"
```

按**最新在前**显示。格式包含：提交哈希、父哈希（空格分隔）、标题。

父哈希有 2 个及以上表示**合并提交**。

`--first-parent` 确保只显示当前分支自身的提交 ——

通过合并引入的提交（如 merge main）**不展开**、不单独显示。

合并点本身会作为**单个合并提交**出现在列表中，并在步骤 3 中原样保留。

如果范围为空，提示用户并停止。

## 步骤 2：分析与提议

使用步骤 1 的 `git log` 输出进行分析：

### 分析流程

**严禁**为了分组而使用 `git diff`，也**严禁**对单个提交使用 `git diff`。

唯一允许的用法是在**第 2 轮**：每组执行一次 `git diff`，读取整体差异用于撰写新提交信息。

步骤 2 期间不得运行其他 Bash 命令。分组所需信息全部来自步骤 1 的 `git log`（提交信息与 `--stat` 文件列表）。

**预处理 —— 标记合并提交**

识别所有合并提交（有 2 个及以上父提交）。

立即标记为 **[merge]** —— 它们在步骤 3 中原样保留，不再参与分析。

合并提交作为**分段边界**，将其余提交分割为独立片段。

**第 1 轮 —— 扫描并分组**

在每个片段内（两个相邻合并提交之间，或范围边界与合并提交之间的非合并提交），

按下面的**分组规则**判断哪些相邻提交可以合并。

**第 2 轮 —— 撰写提交信息**

对每个合并或重写的组，读取**整组的整体差异**：

plaintext

```
git diff <最旧提交>~1 <最新提交>
```

使用 `<最旧提交>~1` 确保包含最旧提交的修改。

每组只做一次 diff，而非每个提交一次。

---

### 分组规则

逻辑很简单：**按顺序遍历提交，判断相邻两个是否可合并**。

**绝不重排提交**。

例如：提交 A 和 C 是互逆操作，但中间有 B，**不要**试图去掉 A 和 C，

只需正常判断 A↔B、B↔C 是否相邻可合并。

对每一对相邻非合并提交，收集它们 `--stat` 的文件列表，计算：

**重叠比例 = 共享路径数量 / 较小列表的大小**

满足任一条件即可合并：

1. **文件重叠 ≥ 50%** —— 提交修改的文件大部分相同，说明属于同一功能模块。
2. **提交信息相似** —— 文件重叠不足 50% 或无重叠，但信息明显指向同一模块 / 功能关键词
    
    （如都提到 “TextBox”“squash 脚本”“CMake option”），
    
    或后一条是延续性描述（“fix…”“wip”“update”“clarify”）。

一旦两个提交形成一组，将下一个提交与**整组所有文件的并集**比较，

用同样的重叠比例与信息相似度判断，**贪心扩展组**。

以下情况**不合并**：

- 两个条件都不满足 —— 文件无交集 **且** 信息指向不同功能。
- 提交被合并提交隔开。

---

### 提交信息质量检查

对照项目提交规范，检查所有未分组的提交信息。

标记需要重写的信息（如模糊的 “wip”“fix”“update”，或不符合格式规范）。

---

### 结果

如果无可合并、无可重写内容，告知用户历史已整洁，并**直接停止 —— 不展示报告、不提问**。

否则，为每个提交标记：

**[keep] 保留 / [squash] 合并 / [reword] 重写 / [merge] 合并提交**

并生成合并报告（暂不展示）。

---

### 报告格式示例

plaintext

```
合并报告（7 条提交 → 4 条提交）：

1. ccc3333 为列表接口添加分页支持。

2. 更新 API 错误响应格式。
   - bbb2222 wip

3. 添加带单元测试的输入验证。
   - def5678 为登录表单添加输入验证。
   - 789abcd 为输入验证添加单元测试。
   - aaa1111 修复验证测试中的断言。

4. abc1234 添加用户认证模块。
```

按**时间倒序**排列（最新在前，最旧在后），

与 `git log` 输出和第 1 轮分析顺序一致。

未改动的提交显示原哈希与信息。

合并 / 重写条目显示**提议的新信息**，下方列出原提交。

报告**只覆盖**分析范围内的提交。

此处**不展示**报告 —— 报告会在 B 阶段展示并让用户确认。

## 步骤 3：执行与替换

### A 阶段 —— 在临时工作区构建新历史

创建工作区前，记录原分支的上游追踪分支（如通过 `git config branch.<name>.remote` 和 `git config branch.<name>.merge`），以便在 C 阶段恢复。

使用 `git worktree add` 在 `{squash_base}` 创建**临时工作区**与**临时分支**。

分支与目录使用固定格式：`squash-tmp-{分支名}`。

下面所有操作都**在这个临时工作区内部执行** ——

用户的工作区、暂存区、原分支在 C 阶段前完全不动。

按**步骤 1 列表**（不是原始 git 范围）的**时间正序**处理提交（最旧在前，与步骤 1 顺序相反）：

- **[keep]**：`git cherry-pick --allow-empty`。连续 [keep] 可批量合并为一条 `git cherry-pick A B C ...`。
- **[reword]**：`git cherry-pick --allow-empty`，然后 amend 修改信息。
- **[squash] 组**：记录当前 HEAD 为 `<saved_head>`，
    
    然后**逐个** cherry-pick 组内提交：
    
    `git cherry-pick --allow-empty <hash>`。
    
    全部 pick 完后：
    
    `git reset --soft <saved_head>` 将它们折叠到暂存区，
    
    再 `git commit -m "新信息"` 生成一条合并提交。
- **[merge]**：必须用 `git commit-tree`（cherry-pick 无法创建合并提交）。
    
    传入原合并提交的 tree、工作区 HEAD 作为第一父节点（`-p`）、原合并提交的第二父节点作为第二父节点（`-p`）。
    
    然后 `git update-ref HEAD <新哈希>`，再 `git reset --hard HEAD` 同步工作区。

任意一步失败：删除临时工作区与分支，提示用户并停止。

**完整性检查**：

比较 `{squash_end}` 与临时分支 HEAD 的 tree，必须完全一致。不一致则中止（删除工作区与分支）。

接下来，记录原分支当前 HEAD 为 `{current_head}`。

如果 `{current_head} ≠ {squash_end}`，说明分析范围之后还有新提交。

在**工作区内部**将这些新提交（`{squash_end}..{current_head}`）原样 cherry-pick 到临时分支。

---

### B 阶段 —— 停止并获取用户确认

**这是强制交互点，不得跳过。**

校验：`git diff {current_head} <临时分支HEAD>` 必须为空 ——

最终文件树必须与 `{current_head}` 完全相同。否则中止（删除工作区与分支）。

在修改原分支前，**必须**完成：

1. 展示合并报告（使用步骤 2 格式），并显示合并前后提交数量。
2. 调用 `AskUserQuestion`，提供 **确认 / 丢弃** 选项。
3. **等待用户回复**。在用户回复前**不得调用任何工具**。

如果用户选择 **丢弃**：删除临时工作区与分支，停止。

仅当用户选择 **确认** 后，才进入 C 阶段。

用户确认后，再次检查是否有新提交：

比较原分支当前 HEAD 与 `{current_head}`。

若不同，将新提交在工作区内 cherry-pick 到临时分支，更新 `{current_head}`，重新校验 tree 差异。

循环直到 `{current_head}` 稳定且校验通过。

---

### C 阶段 —— 替换分支（仅在 B 阶段用户确认后执行）

1. 删除临时工作区（临时**分支**仍保留）。
2. 用 `git update-ref` 将原分支指向临时分支的 HEAD（不要用 `git branch -f`，它会拒绝当前分支）。
    
    这只会更新引用 —— 工作区、暂存区、任何未提交修改都**不受影响**。
3. 删除临时分支。
4. 如果原分支原本有上游追踪分支，用 `git branch --set-upstream-to` 恢复。

输出：

`合并完成：{原数量} 条提交 → {新数量} 条提交`
